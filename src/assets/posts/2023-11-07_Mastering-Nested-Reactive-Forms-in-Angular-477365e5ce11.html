Understanding using formGroupName directive inside child components

Reactive Forms in Angular offer powerful tools for building complex and dynamic user interfaces. While it‚Äôs common to use FormGroup within a single parent component, there are scenarios where you may want to manage form logic in child components or even nest child components within child components. In this article, we‚Äôll explore three examples of working with nested Reactive Forms: a single-parent component, a child component, and a child-in-child component.

1. Single-Parent Component

Your form logic is contained within the same component in a single-parent component. Here‚Äôs an example of what that might look like:

// form-group-parent.component.ts 

export class FormGroupParentComponent {
  public form: FormGroup<IUserForm>;

  constructor() {
    this.form = new FormGroup<IUserForm>({
      name: new FormControl('Max'),
      age: new FormControl(20),

      address: new FormGroup<IAddress>({
        country: new FormControl('Poland'),
        city: new FormControl('Wroclaw'),

        contacts: new FormGroup<IContacts>({
          phone: new FormControl('123456789'),
          email: new FormControl('test@test.com'),
        }),

      })
    });
  }
}

It initializes the form property with a new instance of FormGroup. The generic type <IUserForm> provides a structure that defines the expected form data.

And here‚Äôs our new child component:

// child-form-group.component.ts

@Component({
  selector: 'app-child-form-group',
  standalone: true,
  imports: [CommonModule, ReactiveFormsModule, ChildInChildFormGroupComponent],
  templateUrl: './child-form-group.component.html',
  styleUrls: ['./child-form-group.component.scss'],
  viewProviders: [{ provide: ControlContainer, useExisting: FormGroupDirective }]
})
export class ChildFormGroupComponent {}

The main point here is to add the viewProviders configuration to our Angular @Component decorator.

This configuration is used to provide the component with access to the FormGroupDirective, which is required for working with Reactive Forms. The provide property indicates the token or service that should be provided, and useExisting specifies that it should use an existing instance of FormGroupDirective.

3. Child-in-Child Component

Now, let‚Äôs take it a step further and nest another child component within our child component:

// child-in-child-form-group.component.ts

@Component({
  selector: 'app-child-in-child-form-group',
  standalone: true,
  imports: [CommonModule, ReactiveFormsModule],
  templateUrl: './child-in-child-form-group.component.html',
  styleUrls: ['./child-in-child-form-group.component.scss'],
  viewProviders: [{ provide: ControlContainer, useExisting: FormGroupDirective }]
})
export class ChildInChildFormGroupComponent implements OnInit{
  private formGroupDirective = inject(FormGroupDirective);
  public parentForm!: FormGroup;
  public contactsForm!: FormGroup;

  ngOnInit(): void {
    this.parentForm = this.formGroupDirective.form;
    this.contactsForm = this.parentForm.get('address')?.get('contacts') as FormGroup;
  }
}

This piece of code assigns the form property of FormGroupDirective to the parentForm property. This means that the component is accessing and storing a reference to the parent form, likely for interaction or data retrieval. We also obtain a nested FormGroup named ‚Äúcontacts‚Äù within the ‚Äúaddress‚Äù group using the get method.

Next, we have to use the ‚ÄúcontactsForm‚Äù in formGroup directive in our HTML component, without formGroupNamedirective:

<!-- child-in-child-form-group.component.html -->

<div [formGroup]="contactsForm" class="border-style">
        <span class="user-border-title ">
      Contacts info
        </span>

  <div>
    <label for="phone">Phone:</label>
    <input type="text" class="form-control" formControlName="phone" id="phone">
  </div>
  <div>
    <label for="email">Email:</label>
    <input type="text" class="form-control" formControlName="email" id="email">
  </div>

</div>

In this example, we have successfully used a child component within another child component, providing a clear and organized structure for managing complex forms.
